### 关键教训：Scribe 的职责边界

文件探索能恢复约 **70%** 的上下文：
- 项目结构、技术栈 → `ls`, `cat CLAUDE.md`
- 任务内容 → `git diff`, `README.md`
- 关键决策 → `cortex/adr/`, `skill-deck.toml`

Scribe 必须 dump 那 **30% 探索无法恢复** 的部分：
- **踩过的坑**：尝试 A 失败，换成 B 才成功——这个弯路在最终结果里完全看不到
- **真实 working tree 状态**：哪些文件已提交、哪些未提交、哪些只是 artifact
- **具体的下一步**：不是"测试一下"，而是"运行 `bun xxx` 验证 YYY"

### 最危险的幻觉

两个 Phase B agent 都声称：
> "combo 改动在 working tree 中未提交"

**事实**：`git status` 显示零改动。combo 变化只在 `checkpoints/control-phase-a.diff` 和 `memory-phase-a.diff` 中。两个 agent 都把 diff artifact 误读为 live working tree。

### 机制改进

1. **HANDOFF-TEMPLATE.md** 重写为 7 段式单文件：
   - Project Identity / Session Work / Decisions / Pitfalls & Corrections / Ground Truth State / Next Steps / Handoff Checklist
   - Pitfalls 和 Ground Truth State 用 ⭐ 标注为强制填写

2. **project-scribe** 更新：
   - 强调"dump session 专属信息"，不要重复文件探索能恢复的内容
   - 交接前强制三重确认：`git status` → `cortex list` → session recall

3. **project-onboarding** 更新：
   - 优先读取 `daily/HANDOFF.md`，只有 handoff 缺失/stale 时才降级为目录探索
   - 读取后必须验证 Ground Truth（`git status` 对比 handoff 记录）

4. **daily/ 目录** 建立：
   - `daily/HANDOFF.md` — 当前 session（固定文件名）
   - `daily/YYYY-MM-DD.md` — 历史归档（时间戳平铺，无压输入）
   - 本质：项目的非 task 工作手帐，随 repo 版本管理，跨 CLI 可读
