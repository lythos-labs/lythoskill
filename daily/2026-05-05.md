### Bugs 发现 & 修复

| Bug | 位置 | 原因 | 修复 |
|-----|------|------|------|
| task 文件找不到 (ENOENT) | `runner.ts:106` | `resolve(taskPath)` CWD-relative 而非 configDir-relative | → `taskAbs` |
| deck 路径二次 resolve | `runner.ts:110` | `resolve(cell.deck)` 重复 resolve 已绝对化的路径 | → `cell.deck` |
| Per-cell judge criteria 格式 mismatch | `judge.ts` | LLM 输出 `criteria: {k: "PASS"}` (对象) 而非 `criteria: [{name, passed}]` (数组) | 添加 normalizeVerdictJson fallback |
| Comparative judge pivot-table 格式 | `comparative-judge.ts` | LLM 输出 `{bare: {correctness: 5}}` 而非 `{score_matrix: [...]}` | 添加 pivot→flat conversion |

### 架构决策

**Prompt 策略：Zod schema 字面声明 > 自然语言格式描述**

Conversation 中确认的关键洞察：
- Zod 语法在 LLM 训练数据里的密度远高于任意自然语言格式描述
- Agent 时代的 IoC 心态：把 spec（Zod schema）直接给 agent，而非 post-hoc normalize
- 保留 normalization 为 fallback（防御性），但主导策略是 prompt-level spec

两个 judge prompt 都改为直接嵌入 Zod schema literal：
- `comparative-judge.ts` → `z.object({ score_matrix: z.array(z.object({...})), ... })`
- `judge.ts` → `z.object({ verdict: z.enum([...]), reason: z.string(), criteria: z.array(z.object({...})) })`
