## 调研：Audit-fix Task 被 Re-work 的根因分析

> 记录时间：2026-05-03 14:1X

用户指示终止 `TASK-20260430012458517`（add allowed-tools）和 `TASK-20260430012459381`（review reference triggers），原因是"目前不觉得痛"。但在终止前做了源码级调研，发现值得沉淀的 insight：

### 发现 1：Task Card 空模板 = Re-work 的充分条件

两个 task 的背景/目标/需求/验收标准全部是空模板。子 agent 拿到标题只能猜，猜出来的产出 review 时必然对不上预期。

### 发现 2：Allowed-tools 不是所有 Skill 都需要

盘点全部 10 个 skill：
- **有 allowed-tools 的**（4 个）：deck、cortex、arena、curator — 核心操作都是调用特定 npm CLI（`bunx @lythos/...`）
- **没有 allowed-tools 的**（6 个）：release、scribe、creator、onboarding、coach、hello-world — 核心操作是 agent 通用能力（Shell/WriteFile/ReadFile）

**规律**：allowed-tools 只在"需要调用特定外部 CLI"时声明。Agent Skills 规范中它也是可选字段。

Skill-coach 机械扫描"谁没有 allowed-tools" → 把"不需要"误判为"缺陷"。这是 **false positive**。

### 发现 3：Reference Trigger 可能已经全绿

全部 skill 的 reference table 已使用 "When you need to… | Read" 条件格式，未发现 "See references/ for more details" 模糊触发。Task 的"22 个 references 需要 review"前提可能不存在。

### 决策

- 两个 task → **terminated**（用户："不痛就不治"）
- 调研结论 → 写入 wiki lessons（`cortex/wiki/03-lessons/skill-audit-false-positive-and-task-card-hygiene.md`）

### Meta-lesson

Audit 工具（skill-coach / curator audit）的扫描规则如果未经"是否需要"判断，会产生 false positive。Task 创建者如果没有把 audit 原始报告的具体发现写入 task card，子 agent 只能按标题字面执行，结果往往是"治了不该治的病"。
