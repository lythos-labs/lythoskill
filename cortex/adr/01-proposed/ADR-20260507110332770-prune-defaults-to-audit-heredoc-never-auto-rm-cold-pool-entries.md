## 背景

`packages/lythoskill-deck/src/prune.ts` 当前的 `deck prune` 命令:

1. `buildPrunePlan(deckRaw)` 扫描 cold pool,识别"未被本 deck 引用的 repo"作为 candidates
2. 默认 interactive prompt 让用户确认
3. `--yes` flag 跳过 prompt,直接 `rmSync(repoPath, { recursive: true, force: true })`

`buildPrunePlan` 的实现(`prune-plan.ts:113-132`)只查**单个** deck.toml 的 declared paths——多 workspace 共享同一 cold pool 时,从 workspace A 跑 prune 会把 workspace B 在用的 repo 标为 unreferenced。

### 触发链(本次决策的真实推动)

- 2026-05-07 早晨事件:agent 直接操作 cold pool 状态(覆盖 `skill-deck.toml`),无 service gate 拦截
- 同日 `EPIC-20260507020846020` 推进时,曾计划"加 metadata sidecar 做 ref-counting,prune 时检查 referencing-decks 列表"作为 fix
- 用户 2026-05-07 反馈:"prune 因为太过危险,我真的建议只是帮忙生成带 rm 的 heredoc 算了,一定要审计,不能跳过的那种"
- 同日补充:"如果搞错了状态,非常容易错上加错"——指向 ref-counting 自身也可能弄错事实(metadata 漂移、未注册的新 deck 等)

历史 E2E 失败记录:用户记得 prune 是过往 e2e 超时/exit 的元凶之一("这俩都是之前 e2e 超时/exit 过的玩意,我有印象的")。

## 决策驱动

1. **destructive 操作的爆炸半径不可由 agent 单点决策**——cold pool 是跨 workspace 共享资源,本质上 agent 不掌握全局信息
2. **"有 plan 不等于授权执行"**——plan 是 belief snapshot(扫描时刻的事实),执行要求 belief 仍然 hold;cold pool 是共享资源,几秒钟内可能就有另一个 workspace 在 link 新条目。用户原话:"为哈我们已经画了 plan 图,还不准自动呢"——因为画图本身只证明"agent 看到了什么",不证明"现在还可以这么做"
3. **ref-counting 化解的是 symptom,不是 root cause**——只要"agent 自动 rm"这条路径存在,就会有 sleep 用户/急切 agent 在错误判断下触发删除。让自动 rm 不存在,才彻底
4. **tar 备份不是出路**——加 tar-before-rm 看似安全,但 prune 的存在理由是"回收空间"。tar 会把"腾出来的空间"立刻被备份占满(甚至更大,因为 tar 不去 .git pack),功能定位自相矛盾。用户原话:"如果 tar 一份方便 rollback,这个功能本身腾空间的定位就很可笑,还不如不提供"
5. **审计-就绪 heredoc 是 cost-asymmetric 设计**——agent 渲染 + user 审计的总成本远小于"删错后从 git remote 重新 clone 全部 cold pool"的成本。此外 heredoc 本身**保留了用户决策权**(per `ADR-20260507021957847` ColdPool manager 不变量)
6. **"变成这样的理由"必须显化**——commented-out `# kept: ... referenced by ...` 行让用户看见**为什么没被删的也写在脚本里**,可以反向 sanity-check 边界检测本身是否 buggy(per user 2026-05-07: "如果没有和 link 做好招呼这一步非常容易出 bug 的吧")
7. **0.10.0 release window 允许破坏性变更**——可以删 `--yes` flag、删 interactive confirm,不必保留向后兼容

## 选项

### 方案 A: 现状 + 加 metadata sidecar 做 ref-counting — Rejected

每个 cold pool 条目带 `.cold-pool-meta.json`,记录 referencing-decks 列表。`deck add` 注册引用,`deck remove` 减引用,`prune` 删除时检查 refcount=0 才动手。

- **优点**: 保持自动化 UX 不变,用 metadata 守门
- **缺点**:
  - 仍然是"agent 单点决策删除"——sidecar 漂移、未注册的 workspace、文件系统手工修改都会让 metadata 失真
  - 失真时 agent 会"理直气壮删错"——错上加错,因为 agent 现在确信自己有 metadata 支撑
  - 增加 agent 理解负担(metadata sidecar 是另一个状态源)和 deck 实现复杂度
  - 用户原话:"如果事实错误,是灾难引入器"——ref-counting 把 metadata 升级为 fact source,但 fact 错的概率不为零

### 方案 B: Heredoc rendering with audit annotations — Selected

`deck prune` 输出一个 shell 脚本,而不是直接执行删除。脚本包含两类行:

```bash
#!/bin/sh
# Generated by deck prune at 2026-05-07T11:30:00
# Audit before running. Use deck validate-prune to reality-check first.

# rm -rf for entries deemed unreferenced (size annotated):
rm -rf '~/.agents/skill-repos/github.com/foo/abandoned'   # 12.3 MB

# kept: github.com/anthropics/skills (referenced by ./skill-deck.toml [tool.skills.pdf])
# kept: github.com/lythos-labs/lythoskill (referenced by ./skill-deck.toml [innate.skills.lythoskill-deck])
```

User 打开脚本,审计每行 rm,可以注释掉想保留的。手工执行 `bash prune.sh`。

- **优点**:
  - 删除决策权回到 user 手里,与 `ADR-20260507021957847` 的 "manager 是 user 的 lock 文件" 立场对齐
  - `# kept: ... referenced by ...` 注释暴露**未删的依据**——user 反向看也能 sanity-check 检测算法是否 buggy
  - 不需要 metadata sidecar,deck 实现更简单
  - 与 `ADR-20260507014124191` 的 plan-as-data 立场自洽:plan 即审计报告,执行决策在 user
- **缺点**:
  - User 需要手工执行一步——但本来就该手工(prune 是低频 + 高风险,不该被脚本化兜底)
  - heredoc 自身的事实声明(谁 referenced 了什么)如果错,user 会被误导——通过 `validate-prune` companion 化解(见 `ADR-20260507110332831`)

### 方案 C: 移除 prune 命令,让用户手工 `rm -rf` — Rejected

完全不提供 prune,全靠用户自查 cold pool 大小自己清理。

- **优点**: 极简
- **缺点**: 用户不知道哪些是不再被 deck 引用的,失去 deck 提供的扫描+识别价值——而扫描+识别的部分是 deck 比手工有优势的地方,只是**执行**部分不该 deck 担

### 方案 D: 现状 + tar-backup-before-rm 兜底 — Rejected

执行 rm 前先 `tar czf .claude/prune-backup.tar.gz <candidates>`,允许 rollback。

- **优点**: 万一删错可以解压恢复
- **缺点**:
  - **功能定位自相矛盾**——prune 的存在理由是"回收 cold pool 空间",tar 一份就把腾出来的空间立刻吃回去(实际更糟,git pack 在 tar 后体积可能更大)
  - 只是把"立即灾难"换成"延后灾难"(用户忘了删 tar、tar 滚到磁盘满、tar 自己 corrupt)
  - 用户原话:"如果 tar 一份方便 rollback,这个功能本身腾空间的定位就很可笑,还不如不提供"
  - 注:deck link 的 `ADR-20260502010100000` 是 tar backup 在 working set(`.claude/skills/`)的运用,**那里 tar 是合理的**(working set 不是为腾空间存在);prune 不一样

## 决策

**选择**: 方案 B — heredoc rendering with audit annotations。

**原因**:

1. 保留 deck 的扫描/识别价值(用户手工没法做)
2. 把执行决策权交还 user,符合 ColdPool manager + user-authority 立场(`ADR-20260507021957847`)
3. 与 plan-as-data pattern 对齐(`ADR-20260507014124191`),输出是数据/脚本不是动作
4. annotations 让边界检测自身可被 user 反向 sanity-check
5. 配合 `validate-prune` companion 把"事实错误"风险压到极低(`ADR-20260507110332831`)
6. 0.10.0 窗口允许去掉 `--yes` 这个 footgun-flag

## 影响

### 正面
- 不会再发生"agent 自动删错 cold pool 条目"类灾难
- `deck prune` 实现简化(无需 metadata sidecar / interactive prompt / rmSync IO)
- Cold-pool executor 层不需要导出 `rmRf` IO,executor cannot delete 的不变量明示
- 和 `cold-pool reconcile` 命令的"remove orphans"路径用同一规则,统一 UX

### 负面
- 用户多一步手工 `bash prune.sh`——但 prune 本身是低频操作,可接受
- Heredoc 自身可能含错误 fact 声明——通过 `validate-prune` companion 兜底
- 删除 `--yes` 是破坏性变更——0.10.0 release 时一并

### 后续
1. 实施 `deck prune` 重构(removes `--yes`、interactive prompt、`rmSync` 调用)
2. 实施 `deck validate-prune <plan-path>` companion(per `ADR-20260507110332831`)
3. `cold-pool reconcile` 命令的 "remove orphans" 路径同样产 heredoc(后续 epic)
4. README + AGENTS.md 更新 prune 用法(0.10.0 release notes)
