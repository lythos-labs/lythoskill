# ADR-20260501160000000: skill-deck.toml section semantics and innate skill re-attachment after context compaction

## Status History

| Status | Date | Note |
|--------|------|------|
| proposed | 2026-05-01 | Created — pending implementation |

## 背景

`skill-deck.toml` 有四个 section：`innate`、`tool`、`combo`、`transient`。当前实现中，这四个 section 除了打 `type` 标签存入 lock 文件外，没有任何行为差异。

同时，Agent Skills 平台有一个隐式约束：当 agent context 被 compacted 后，skill 会以 **5,000 tokens 截断摘要** 的形式重新 attach。对于需要 agent 完整理解的 skill，截断意味着失效。

两个问题需要同时回答：
1. 四个 section 的行为差异应该是什么？
2. `innate` skill 如何保证 compaction 后仍是"完整在场"？

## 决策驱动

- **预算归属**：`innate` 和 `tool` 必须有不同的预算语义，否则区分无意义
- **TCG 类比**：`innate` = 被动技能（不占卡槽），`tool` = 主动卡牌（占卡槽）
- **编排层识别**：`combo` skill 是调度者而非实现者，agent 需要知道这层区别
- **TTL 语义**：`transient` 是 workaround，必须有明确的生存周期
- **compaction 免疫**：`innate` skill 的完整内容必须在每次 context 初始化时可达

## 问题一：Section 行为差异

### 选项 A：保持现状（纯语义标签）

四个 section 只是标签，无行为差异。

**优点**：
- 实现最简单
- 未来可扩展

**缺点**：
- 认知成本 > 收益——用户学了四个概念，却没有任何功能回报
- `innate` 和 `tool` 的区分沦为"你觉得哪个好听就用哪个"

### 选项 B：差异语义 + 差异行为（Selected）

| Section | 行为差异 |
|---------|---------|
| `innate` | 不计入 `max_cards`；通过 `INNATE.md` inline 保证 compaction 后完整可读 |
| `tool` | 计入 `max_cards`；受 deny-by-default 管理 |
| `combo` | 计入 `max_cards`；命名强制 `combo-` 前缀；SKILL.md 必须包含触发条件和路由规则 |
| `transient` | 计入 `max_cards`；`deck link` 自动清除过期条目 |

**优点**：
- 每个 section 的区分都有明确的功能回报
- 符合 TCG 类比（`innate` = 被动不占卡槽）
- `combo` 的命名约定让编排 skill 一眼可识别

**缺点**：
- 实现复杂度更高
- 需要处理 `INNATE.md` 的生成和维护

## 问题二：`innate` skill 的 compaction 免疫

### 核心问题

Agent Skills 平台的 compaction 机制：当 context window 接近上限时，skill 内容会被截断为 5,000 tokens 的摘要。Compaction 后 agent 重新入驻时，只能看到摘要，看不到全文。

对于 `innate` skill（如 `lythoskill-deck` 本身），摘要意味着失效——agent 需要完整理解 deny-by-default 的治理规则，不能只靠"大概记得"。

### 角度 1：`INNATE.md` inline（Selected）

`deck link` 生成 `.claude/INNATE.md`，将 `innate` section 所有 skill 的 SKILL.md 全文 inline 进去。用户在 `CLAUDE.md` 顶部加一行引用：

```markdown
@include .claude/INNATE.md
```

`CLAUDE.md` 是 agent 每次 context 初始化必读的，且不受 5,000 tokens 截断限制（它是项目文件，不是 skill）。`INNATE.md` 作为 `CLAUDE.md` 的一部分被完整读取。

**优点**：
- `innate` 语义和实现完全一致：用户说"这是基础能力"，工具回答"好，我把它放到 agent 必读位置"
- 零 agent 自律要求，不依赖启发式
- 跨 compaction 周期稳定

**缺点**：
- `INNATE.md` 可能很大，但这是用户显式选择的 context 成本
- `deck link` 需要读写 `INNATE.md`
- `@include` 语法是 Claude Code 特有，其他 CLI 可能需要适配

### 角度 2：`when_to_use` 自律提示

在 `innate` skill 的 `when_to_use` 字段里加：

```
If you are unsure of the full content of this skill after context compaction, re-read SKILL.md before proceeding.
```

**优点**：实现成本极低

**缺点**：依赖 agent 自律，可靠性不足

### 角度 3：`deck link` 生成 summary 注入

`deck link` 生成 `_innate-summary.md`，包含每个 innate skill 的 frontmatter + 关键内容摘要，通过 `CLAUDE.md` 引用。

**优点**：文件体积可控

**缺点**：summary 不是全文，对需要精确规则的 skill（如 deck 的 deny-by-default 逻辑）可能不够

## 决策

**选择**：
1. Section 行为差异选 **方案 B**
2. `innate` compaction 免疫选 **角度 1**（`INNATE.md` inline）

**原因**：

1. **预算豁免是 `innate` 的唯一正确定义**

   如果 `innate` 也计入 `max_cards`，它和 `tool` 就没有任何实质区别。TCG 类比中，被动技能不占卡槽是核心特征——`lythoskill-deck` 作为治理工具永远在场，不应该被 "max_cards = 10" 挤掉。

2. **`INNATE.md` inline 是对用户声明的忠实执行**

   用户把 skill 放进 `innate` section，就是在说"我愿意为它付出 context 成本"。`deck link` 把它 inline 到 `CLAUDE.md` 的可读区域，是对这个声明的机械执行，不需要额外的启发式或 agent 自律。

3. **combo 命名约定降低认知负担**

   `combo-` 前缀让 agent 和人类都能立即识别"这是编排 skill，不是工具 skill"。这比在文档里写"combo skill 的特点是..."更有效。

4. **transient 的 TTL 是最小可行机制**

   `deck link` 已经有 `expires` 检查和警告，只需要加一步自动清理。这是最容易实现的行为差异。

## 影响

### 正面

- `innate` skill 在 compaction 后仍完整可读
- `max_cards` 只约束 `tool` + `combo` + `transient`，`innate` 作为基础设施不计入
- `combo-` 前缀让编排 skill 一眼可识别
- 过期 transient 自动清理，减少幽灵 skill

### 负面

- `INNATE.md` 可能很大，但这是用户显式选择的成本
- `deck link` 需要管理 `INNATE.md` 的生成和更新
- `@include` 语法跨 CLI 不通用，可能需要平台适配层
- `innate` 不计入 `max_cards` 需要防滥用（建议加 `max_innate` 次级约束）

### 风险

- **innate 滥用**：用户可能把大量 skill 放进 `innate` 绕过 `max_cards`。缓解：`max_innate = 3` 硬约束，或 `innate` 数量超过阈值时发出警告。
- **INNATE.md 漂移**：如果 `deck link` 没有正确更新 `INNATE.md`，可能导致内容过期。缓解：`INNATE.md` 头部加 fence 标记，如 `<!-- Generated by lythoskill-deck, do not edit -->`。

## 实施

### Phase 1：预算归属

`link.ts` 中修改预算检查逻辑：

```typescript
// 只计数 tool + combo + transient
const budgetedCount = declared.filter(d => d.type !== "innate").length;
if (budgetedCount > MAX_CARDS) {
  console.error(`❌ Budget exceeded: ${budgetedCount} budgeted skills, max ${MAX_CARDS}`);
  console.error(`   (innate skills are exempt from max_cards)`);
  process.exit(1);
}
```

### Phase 2：`INNATE.md` 生成

`link.ts` 中在 reconcile 完成后：

```typescript
const innateSkills = declared.filter(d => d.type === "innate");
if (innateSkills.length > 0) {
  const innateContent = innateSkills.map(s => {
    const skillMd = readFileSync(join(s.sourcePath, "SKILL.md"), "utf-8");
    return `<!-- ${s.name} -->\n${skillMd}`;
  }).join("\n\n---\n\n");
  writeFileSync(
    join(PROJECT_DIR, ".claude", "INNATE.md"),
    `<!-- Generated by lythoskill-deck. Do not edit manually. -->\n\n${innateContent}`
  );
}
```

### Phase 3：`transient` TTL 自动清理

在 transient 过期检查（已有）后，加清理逻辑：

```typescript
for (const s of linkedSkills) {
  if (s.type !== "transient" || !s.expires) continue;
  const days = Math.ceil((new Date(s.expires).getTime() - Date.now()) / 86400000);
  if (days <= 0) {
    // 从 working set 移除 symlink
    const wsPath = join(WORKING_SET, s.name);
    rmSync(wsPath, { recursive: true, force: true });
    console.log(`  🗑️  Expired transient removed: ${s.name}`);
    // 可选：从 skill-deck.toml 移除声明（需要 TOML 编辑逻辑）
  }
}
```

### Phase 4：`combo` 命名检查

ADR-check 脚本中加一条：

```bash
# combo section 中的 skill 必须有 combo- 前缀
echo "[ADR-20260501160000000] combo skills must have combo- prefix"
```
