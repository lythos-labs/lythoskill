## Context

在 `lythoskill` 自举过程中，`starterCli` 模板生成了如下代码来读取 package.json：

```typescript
const pkg = require('../package.json')
```

但生成的 starter 设置了 `"type": "module"`，这是一个 ESM-only 包。在纯 ESM 环境中，`require` 会抛出 `ReferenceError: require is not defined`。

## Options Considered

### Option A: 保持 CJS（Rejected）
将 starter 改为 `"type": "commonjs"` 或使用 `.cjs` 扩展名。

- **Pros**: `require` 可用
- **Cons**: 与现代前端生态脱节；Bun/Node 都优先 ESM；失去 top-level await 等特性

### Option B: `createRequire`（Rejected）
使用 `import { createRequire } from 'node:module'` 在 ESM 中构造 require。

- **Pros**: 兼容现有代码
- **Cons**: hacky；不必要的 indirection；违背 ESM 语义

### Option C: `import with { type: "json" }`（Selected）
使用标准的 ESM JSON import assertion：

```typescript
import pkg from '../package.json' with { type: 'json' }
```

- **Pros**: 标准 ESM 语法；Bun/Node 都支持；类型干净
- **Cons**: 需要 tsconfig 中 `moduleResolution: "bundler"`（已有）

### Option D: `Bun.file().json()`（备选）
使用 Bun 特有的 API：

```typescript
const pkg = await Bun.file('../package.json').json()
```

- **Pros**: Bun-native，不需要 import assertion
- **Cons**: 不是标准 JavaScript；仅限 Bun 运行时

## Decision

采用 Option C（`import with { type: "json" }`），因为 lythoskill 明确目标 Bun 运行时，但保持标准 JavaScript 语法以确保未来兼容性。
